# Project Plan: AI Adaptive Fitness Coach Backend

This document contains the functional specifications, database architecture, Java (Spring Boot) class structure, and Artificial Intelligence integration strategy for the **Coach AI** application. The system is designed to generate highly personalized and adaptive workout plans based on continuous user feedback.

---

## PART 1: Project Specifications & Requirements

### 1.1 System Objective
The backend acts as an intelligent, automated Personal Trainer. Instead of distributing static, pre-compiled workout routines, it dynamically composes a custom workout session in real time by intersecting the user's physical profile, logistic constraints, and historical feedback data.

### 1.2 Adaptive Factors (Generation Constraints)
The application computes the optimal workout plan based on 8 core pillars:

* **Training Location:** Immediate adaptation between Home (`HOME`) or Gym (`GYM`).
* **Available Equipment:** Dynamic filtering based on an array of equipment owned by the user (e.g., bodyweight, dumbbells, resistance bands).
* **Difficulty Rating (Level 1-5):** An intensity scale that directly affects total volume (sets/reps) and exercise variation complexity.
* **Physical Limitations & Needs:** Handling of injuries, chronic pain, or disabilities (e.g., "right wrist pain", "avoid jumping"). The AI must exclude or substitute critical exercises.
* **Time Availability:** Management of session duration (e.g., 30, 45, 60 minutes). The system calculates the sustainable number of exercises, factoring in target rest periods and the user's customized pace.
* **Weekly Frequency:** Number of training days requested per week (e.g., 3 or 4 times a week).
* **Body Part Rotation (Split Logic):** Automated muscle group management (e.g., if the last generated workout was *Upper Body*, the next will automatically target *Lower Body* or *Core*).
* **Anti-Boredom Variation:** Rotation of exercises based on recent history to eliminate monotony and maintain high user engagement.

### 1.3 The Feedback Loop
At the end of a workout session, the user rates each individual exercise performed:
* **Effort Assessment:** `TOO_EASY`, `PERFECT`, or `TOO_HARD`.
* **Pain Flag:** A boolean flag indicating unexpected joint or acute muscle discomfort.
* **User Notes:** Open text comments for qualitative feedback.

This state is persisted in the database and fed into the LLM during the next generation request to recalibrate intensity or swap out problematic movements.

## PART 2: Database Structure (PostgreSQL)

The database utilizes the `snake_case` naming convention. Relationships are strictly enforced via Foreign Key constraints. The `JSONB` data type is used for flexible array storage (such as the user's equipment list).

```
  [user_profile] ───1:N───> [workout_plan] ───1:N───> [workout_exercise]
        │                        │                           │
        │                        └───1:N───┐                 │
        │                                  ▼                 ▼
        └───────────────────────────────1:N───> [workout_feedback] <───N:1─── [exercise_catalog]
```

### 2.1 DDL Schema

```sql
-- 1. User Profile
CREATE TABLE user_profile (
    id BIGSERIAL PRIMARY KEY,
    user_id VARCHAR(100) UNIQUE,          -- Unique client identifier (e.g., user ID or Email)
    fitness_level INT NOT NULL,               -- Value from 1 to 5
    training_days_per_week INT NOT NULL,       -- Weekly frequency (2-7)
    available_time_minutes INT NOT NULL,       -- Single session duration limit
    speed_modifier DECIMAL(3,2) DEFAULT 1.00,  -- Multiplier adjusting estimated time per exercise based on user pace
    equipment JSONB NOT NULL,                 -- JSON Array (e.g., ["dumbbells", "kettlebell", "none"])
    physical_limitations TEXT,                -- Open text for injuries/medical conditions
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. Global Exercise Catalog (Baseline Lookup)
CREATE TABLE exercise_catalog (
    id VARCHAR(100) PRIMARY KEY,              -- Kebab-case string ID (e.g., "diamond-push-up")
    name VARCHAR(150) NOT NULL,               -- Human-readable name (e.g., "Diamond Push-up")
    muscle_group VARCHAR(50) NOT NULL,        -- UPPER_BODY, LOWER_BODY, CORE, FULL_BODY
    equipment_needed VARCHAR(50) NOT NULL     -- BODYWEIGHT, DUMBBELLS, BANDS, BARBELL, etc.
);

-- 3. Workout Plan Header
CREATE TABLE workout_plan (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES user_profile(id) ON DELETE CASCADE,
    name VARCHAR(150) NOT NULL,               -- Session title (e.g., "Push Day - Session 1")
    split_type VARCHAR(50) NOT NULL,          -- UPPER_BODY, LOWER_BODY, CORE, etc.
    motivational_message TEXT,                -- Empathetic message generated by Gemma
    difficulty_rating_request INT NOT NULL,   -- Configured difficulty level (1-5) at generation time
    generated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 4. Workout Exercise Details (AI Composed)
CREATE TABLE workout_exercise (
    id BIGSERIAL PRIMARY KEY,
    workout_id BIGINT REFERENCES workout_plan(id) ON DELETE CASCADE,
    exercise_id VARCHAR(100) REFERENCES exercise_catalog(id),
    sets INT NOT NULL,
    reps INT NOT NULL,
    rest_seconds INT NOT NULL
);

-- 5. Workout Performance & Difficulty Feedback History
CREATE TABLE workout_feedback (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES user_profile(id) ON DELETE CASCADE,
    workout_id BIGINT REFERENCES workout_plan(id) ON DELETE CASCADE,
    exercise_id VARCHAR(100) REFERENCES exercise_catalog(id),
    difficulty_rating VARCHAR(50) NOT NULL,   -- TOO_EASY, PERFECT, TOO_HARD
    pain_or_discomfort BOOLEAN DEFAULT FALSE, -- True if the exercise caused atypical pain
    user_notes TEXT,                          -- Additional contextual notes from the user
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## PART 3: Java Architecture (Spring Boot)

The backend conforms to a standard **Controller-Service-Repository** layered architecture. The codebase leverages Java 16+ `record` types for immutable DTOs and Spring Data JPA for data persistence.

### 3.1 Package Layout

```text

│   └── AiModelConfig.java
├── controller/             # REST API Endpoints
│   ├── UserController.java
│   └── WorkoutController.java
├── model/                  # JPA Entities (Database Mapping)
│   ├── UserProfile.java
│   ├── ExerciseCatalog.java
│   ├── WorkoutPlan.java
│   ├── WorkoutExercise.java
│   └── WorkoutFeedback.java

│   ├── UserProfileDto.java
│   ├── FeedbackRequestDto.java
│   ├── WorkoutResponseDto.java

├── repository/             # Spring Data JPA Repositories

│   ├── WorkoutPlanRepository.java

    ├── WorkoutService.java
    └── AiRoutingService.java
```

### 3.2 Key Service Responsibilities

#### `WorkoutService.java` (Orchestrator)
Executes deterministic algorithmic steps in Java prior to querying the LLM to minimize input tokens (Context Optimization):
1. **Split Type Resolution:** Queries `WorkoutPlanRepository` to inspect the user's latest session and determines the optimal target muscle group for today's rotation.
2. **Catalog Pre-filtering:** Filters `ExerciseCatalogRepository` to fetch only the exercises matching the user's available equipment.
3. **Memory Extraction:** Extracts recent negative ratings or pain flags via `WorkoutFeedbackRepository`.
4. **Deterministic Level Tuning:** Maps automated adjustments to the profile difficulty rating based on user history:
  * *Level Progression:* If a user rates all exercises in a workout as `TOO_EASY` for 3 consecutive training sessions, `WorkoutService` automatically increments `user_profile.fitness_level` by 1 (Max 5).
  * *Level Regression:* If a user reports `pain_or_discomfort = true` or rates exercises as `TOO_HARD` twice within a single training session, the system automatically decrements `user_profile.fitness_level` by 1 (Min 1) ahead of the next generation cycle.
5. **Prompt Compiling & Saving:** Pipes the filtered datasets to `AiRoutingService`, ingests the structural response payload, builds the JPA entity tree, and saves it to the database.

#### `AiRoutingService.java` (LLM Gatekeeper)
Isolates Spring AI interaction abstractions. Injects environment configurations to handle HTTP/REST networking payloads with upstream AI endpoints.

## PART 4: Artificial Intelligence Strategy & Architecture

The application implements a **Multi-Model Routing** pattern to optimize financial cost, API inference latency, and response accuracy, effectively separating structural processing from creative writing tasks.

### 4.1 Model Configurations

| Role | Model Selected | Provider | Temperature | Objective |
| :--- | :--- | :--- | :--- | :--- |
| **Workout Generation** | `meta-llama/llama-3-8b-instruct` | OpenRouter / Groq | **0.1** (Low) | Deterministic output structure, absolute adherence to the required JSON schema, precise volume math. |
| **Motivational Message** | `google/gemma-2-9b-it` | OpenRouter / Google | **0.7** (Medium-High) | Fluid, empathetic, and natural English prose tailored to the user's current session state. |

### 4.2 Context Optimization (Prompt Pre-filtering)
To bypass context window bloat, token pricing penalties, and response degradation, Java avoids sending the entire global exercise catalog to the LLM. The input prompt contains exclusively a focused, pre-filtered list of relevant movements (typically a subset of 20-40 targeted records).

### 4.3 Prompt Engineering & Target JSON Schema (Llama 3)
The system prompt strictly forces Llama 3 to respond **only** with a valid JSON serialization payload matching the explicit schema configuration.

#### Expected Output Schema

```json
{
  "workout_name": "Name of the workout session (String)",
  "exercises": [
    {
      "id": "kebab-case-exercise-id (Must exactly match one of the provided IDs in the prompt list)",
      "sets": 4,
      "reps": 12,
      "rest_seconds": 90
    }
  ],
  "discovered_exercises": [
    {
      "id": "hallucinated-or-new-kebab-case-id",
      "name": "Human Readable Exercise Name",
      "muscle_group": "UPPER_BODY",
      "equipment_needed": "BODYWEIGHT"
    }
  ]
}
```

#### Runtime Validation & Fallback Rules (Java)
1. **JSON Mode Execution:** Informs the API provider endpoints to actively enforce JSON formatting compliance structural outputs.
2. **Database Crash Fallback Handler:** If Llama hallucinates an exercise ID omitted from the pre-filtered prompt list, Java catches the parsing exception. It checks the `discovered_exercises` array to find the required database metadata (`name`, `muscle_group`, `equipment_needed`).
  * If the metadata block is present, Java handles an *on-the-fly* dynamic write insertion into `exercise_catalog` without violating `NOT NULL` constraints before successfully binding the transaction log.
  * If an ID is unknown and no descriptive metadata block is provided by the LLM, Java discards that single exercise item and logs an internal warning instead of throwing an unhandled database exception and crashing the request thread.
